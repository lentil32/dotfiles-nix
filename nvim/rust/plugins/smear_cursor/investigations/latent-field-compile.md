# Smear Cursor Latent-Field Compile Investigation

Scope: practical per-frame slowdown risk in the current latent deposition and compile pipeline.

## What This Note Covers

This note tracks the render path that is most likely to cause observable slowdown during normal
motion: latent deposition plus full latent-field recompilation.

The key path is:

1. [`compile_render_frame`](../src/draw/render_plan/lifecycle.rs#L90) stages new motion samples.
2. [`stage_deposited_samples`](../src/draw/render_plan/solver.rs#L1396) deposits up to three band
   slices per simulation sample.
3. [`deposit_swept_occupancy`](../src/draw/render/latent_field.rs#L657) rasterizes each swept pose
   into microtiles.
4. [`compile_field_from_cache`](../src/draw/render/latent_field.rs#L878) rescans the retained
   latent cache to rebuild the compiled field.

## Why This Is The Practical Slowdown Risk

The compile path scales with retained live trail history, not just with one slice or one frame's
local neighborhood.

Let:

- `A_i` = occupied microtiles generated by deposited slice `i`
- `H = sum(A_i)` across all live retained slices/buckets in the latent cache
- `U` = unique output cells that survive into the compiled field

Then:

- one deposition step is roughly `O(A_i)` time and `O(A_i)` space
- one compile is effectively `O(H + U)` time, but it scans the retained cache twice and pays a
  64-sample accumulation cost per surviving cell

The two-pass compile is visible in:

- mass accumulation in [`compile_field_from_cache`](../src/draw/render/latent_field.rs#L881)
- sample accumulation in [`compile_field_from_cache`](../src/draw/render/latent_field.rs#L898)

## Why Active Motion Keeps Paying The Cost

The compiled field cache only reuses work when the latent cache revision is unchanged in
[`compiled_field_for_state`](../src/draw/render_plan/lifecycle.rs#L104).

During active motion that condition usually fails because:

- [`insert_slice`](../src/draw/render/latent_field.rs#L365) increments the latent cache revision
- [`compiled_field_for_state`](../src/draw/render_plan/lifecycle.rs#L115) sees the revision drift
  and recompiles

So the plugin can end up rebuilding the compiled latent field every active frame.

## What Makes It Pathological

This path gets worse when any of these increase:

- `simulation_hz`
- `tail_duration_ms`
- `step_samples` emitted in one frame
- cursor travel distance per sample
- trail thickness

Reasons:

- support steps scale with duration and simulation rate in
  [`tail_support_steps`](../src/draw/render/latent_field.rs#L499) and
  [`max_comet_support_steps`](../src/draw/render/latent_field.rs#L649)
- deposition work expands with the swept bounding box in
  [`deposit_swept_occupancy`](../src/draw/render/latent_field.rs#L682)
- every simulation sample is processed in
  [`stage_deposited_samples`](../src/draw/render_plan/solver.rs#L1401)

## Space Cost

The persistent memory cost is also concentrated here.

The retained latent cache stores:

- per-curve/per-age buckets in [`WeightCurveBuckets`](../src/draw/render/latent_field.rs#L246)
- bucket maps under [`LatentFieldCache`](../src/draw/render/latent_field.rs#L318)
- per-cell sample arrays in [`BucketCell`](../src/draw/render/latent_field.rs#L190)

Each retained bucket cell carries a `[u32; 64]` sample sum plus mass bookkeeping, so larger support
windows raise both compile time and retained memory together.

## Comparison With The Ribbon Solver

The ribbon solve is algorithmically more interesting, but it is bounded by tight caps in
[`infra.rs`](../src/draw/render_plan/infra.rs#L561). The latent-field compile path does not have a
similar small constant-state cap; it scales with the actual retained trail footprint and history.

That is why this path is the better answer to "what will probably make the plugin feel slow?"

## Bottom Line

The most likely real-world slowdown comes from latent deposition plus recompilation, centered on
[`compile_field_from_cache`](../src/draw/render/latent_field.rs#L878).

Comment: the rasterizer in [`deposit_swept_occupancy`](../src/draw/render/latent_field.rs#L657)
also pays a full row/column Cartesian product over covered cells, so large diagonal sweeps can look
more expensive than their visible ribbon thickness first suggests.
